# 運用・配信（サブC）設計書

- 作成日: 2026-06-22
- 対象: 既存「G7一次情報＋マクロ数値ダッシュボード」への拡張
- 位置づけ: プロ向け拡張 A→B→C の **サブC（最終）**。A（シグナル）・B（データ拡充）実装済み。

## 1. 目的

豊富になったシグナル/データを「読みやすく・自動で届く」形にする。週次ブリーフィングの自動生成、データ鮮度の自動警告、チャートのホバー値表示。

## 2. スコープ（確定要件）

- C8 週次ブリーフィング：**自動テンプレート**（LLM不使用）。build.py がシグナルから定型文を生成。
- C9 データ鮮度フラグ：観測間隔から自動判定し、古い系列を警告。
- C10 本格チャート：ホバーで日付＋値を表示（依存ライブラリなし）。
- 週次タスク（既存）で全自動更新。Excel・取得処理は無変更。

### スコープ外（YAGNI）
- LLM生成ブリーフィング、外部配信（メール等）、モーダル詳細画面。

## 3. C8 週次ブリーフィング（build.py）

- `build_briefing(series: list, signals: dict, as_of: str) -> dict` → `{"as_of": str, "lines": [str,...]}`。
- 生成する行（空セクションは省略、レジームは常に出力）:
  1. `リスクレジーム: {label} (総合スコア {score})`
  2. `直近の主な変化: ` ＋ movers上位（`▲/▼ {indicator}・{country} (Δz {delta_z})`）
  3. `極端な水準(|Z|≥2): ` ＋ `{indicator}・{country} z{zscore} ({高|低})`（|z|降順）
  4. `ポジション偏り(建玉|Z|≥1.5): ` ＋ COT系列（indicator=="建玉(COT)"）
  5. `データ鮮度: ` 異常なし / `⚠ {names}`（C9のstale系列）
- payloadに `briefing` を載せる。`main()` で `週次ブリーフィング.md` に書き出し（Git追跡＝履歴保持）。

## 4. C9 データ鮮度（build.py）

- `_staleness(rows: list, today: date) -> tuple[int,bool]`:
  - `rows` の日付列から直近の**観測間隔の中央値**(日)を求め、`stale_days = (today - 最新日).days`、
  - `stale = stale_days > median_gap*2.5 + 7`（日次/週次/月次/四半期を自動吸収）。
  - 観測2点未満は `(0, False)`。
- payload各系列に `stale_days`・`stale` を付与。`today` は `date.today()`。

## 5. C10 チャート ホバー（index.html）

- 単一系列スパークライン（`macroCard`）に、ホバーで**日付＋値**を表示。
  - `lineChartSVG` の単一系列描画を、`<div class="spark" data-points="...">` で包み、`offsetX/幅` から最寄り点を算出してツールチップ＋縦ガイド線を表示。
  - `data-points` は**60点に間引いて**埋め込み（DOM肥大回避）。
  - 委譲リスナー（document に1つ）でmousemove/leaveを処理。複数系列チャート（マネーの流れ）は対象外（凡例＋レンジで足りる）。
- Y軸の数値は既存の「範囲(最小〜最大)」キャプションで担保済み（重複追加しない）。

## 6. 表示の変更（index.html）

- `renderOverview`：先頭に「週次ブリーフィング」カード（`briefing.as_of` と `lines`）。サブAで追加した「リスクレジーム」パネルと「直近の主な変化」カードは**ブリーフィングに内包されるため置き換え（削除）**し、重複を避ける。
- `macroCard`：`s.stale` のとき ⚠「データ古い (N日)」バッジを表示。
- チャートにホバー（C10）。

## 7. 変更ファイル
- `build.py`：`_staleness`/`build_briefing`、payloadに briefing・stale 付与、`write_briefing_md`、main で書出。
- `index.html`：ブリーフィングカード、鮮度バッジ、チャートホバー。
- `更新手順.md`：週次ブリーフィングの説明を追記。
- `.gitignore`：（briefing.md は追跡するため変更なし）。

## 8. エラーハンドリング・正確性
- ブリーフィングはデータ由来の定型生成のみ（推測なし）。空セクションは出さない。
- 鮮度判定は観測間隔ベースで頻度差に頑健。データ無し系列は除外。
- ホバーは点が無い/壊れた場合に何も表示しない（安全側）。

## 9. テスト
- `_staleness`: 月次サンプル（最新が35日前→stale False、120日前→True）、四半期サンプル（100日前→False）。
- `build_briefing`: レジーム行・極端(|z|≥2)・鮮度の各行が条件どおり生成されること。
- payload に briefing・stale が含まれること。既存テスト全PASS。
